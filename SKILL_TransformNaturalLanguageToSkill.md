## 01 DEFINING — 核心原则

### 价值观陈述

**任何人都能将自己的价值观想法转化成一个清晰、可验证、跨文化适用的技能。**

这个原则的核心在于：
- 🎯 **民主化**：不仅仅是AI研究员、哲学家才能定义价值观
- 📖 **系统化**：通过五层架构将模糊的想法变成明确的、可验证的技能
- 🌍 **跨越性**：同一个技能能在不同文化中被正确理解和应用

### 为什么这很重要？

当前AI系统的困境：
- 价值观隐含在训练数据中（黑盒）
- 用户无法验证AI遵循哪些原则
- 同一个AI在不同文化中表现不一致

**这个Skill要解决的问题**：
提供一个系统化的方法，让任何想要显式表达价值观的人，都能将其转化成：
- ✅ 清晰明确的原则
- ✅ 具体的、可理解的例子
- ✅ 明确的应用边界
- ✅ 可验证的测试用例
- ✅ 跨文化的适配

## 02 INSTANTIATING — 具体例子

### 案例1：心理咨询师定义"沉默的艺术"

**原始想法**：
> "AI应该知道何时保持沉默，陪伴而不是快速解决问题"

**使用这个Skill的结果**：

```
【DEFINING】
核心原则：识别真实的痛苦，不最小化它

【INSTANTIATING】
例子1：用户说"我很难过" 
       → AI陪伴倾听，而不是给出"5个应对步骤"
例子2：用户说"感觉被孤立"
       → AI反映用户的感受，而不是快速正面化
例子3：用户述说失败经历
       → AI承认失败的价值，而不是立即鼓励

【FENCING】
应用场景：心理咨询、情感支持、深层倾听
不应用：日常闲聊、技术问题、快速查询

【VALIDATING】
测试1：用户表达痛苦时，AI没有给出教学性建议？✓
测试2：AI的回应长度不超过一个段落？✓
测试3：AI使用了反映（mirroring）技巧？✓

【CONTEXTUALIZING】
中文：强调"陪伴"和"见证"的价值
日文：强调"間"（空白）和"寂しさ"（孤寂）
英文：强调"presence"（在场）和"witnessing"（见证）
```

### 案例2：设计师定义"审美原则"

**原始想法**：
> "设计应该尊重用户的认知能力，不要过度装饰"

**转化结果**：
```
【DEFINING】
核心原则：清晰性优于华丽，可用性优于视觉冲击

【INSTANTIATING】
好例子：Apple的极简设计 → 用户立即明白功能
坏例子：过度动画的网站 → 用户需要花时间理解页面

【FENCING】
应用于：UI/UX设计、信息架构
不应用于：艺术创作、实验性设计

【VALIDATING】
首次使用的用户是否能在30秒内完成主要任务？
是否有用户测试支持这个原则？
```

### 案例3：工程师定义"代码质量标准"

**原始想法**：
> "代码应该为人类而写，而不仅仅为机器"

**转化结果**：
```
【DEFINING】
核心原则：可读性和可维护性是代码的第一优先级

【INSTANTIATING】
好代码：变量名清晰 + 注释解释"为什么"而不是"是什么"
坏代码：单字母变量 + 复杂的单行逻辑

【FENCING】
应用于：生产代码、团队项目、长期维护代码
不应用于：原型、竞赛代码、一次性脚本

【VALIDATING】
新员工是否能在30分钟内理解这段代码？
是否有测试覆盖这个模块？
```

## 03 FENCING — 应用边界

### 什么时候应该使用这个Skill？

✅ **适用场景**：
1. **价值观模糊**：你有一个想法，但不知道如何清晰表达
2. **跨团队应用**：你需要确保不同背景的人理解相同的原则
3. **需要验证**：你想要一个客观的方式来验证是否遵循了原则
4. **文化适配**：你的价值观需要在不同文化中应用
5. **构建信任**：你想透明地展示你的价值观

### 什么时候不应该使用这个Skill？

❌ **不适用场景**：
1. **已有规则文档**：如果你已经有明确的规则/政策文档，直接用那个
2. **时间紧急**：如果需要立即做决定，五层转化会花时间
3. **完全主观**：如果价值观在不同文化中有根本性冲突，难以调和
4. **法律/合规**：需要律师/专家审核的法律问题不能只靠这个
5. **暂时性决定**：一次性的临时决定不需要五层化

### 潜在的冲突和权衡

**冲突1：清晰性 vs 灵活性**
- 五层架构追求清晰性，但可能牺牲灵活性
- 权衡：接受你的原则可能不适用于所有边界情况

**冲突2：跨文化 vs 本地化**
- CONTEXTUALIZING层尝试跨文化适配，但有些概念本身就无法翻译
- 权衡：接受某些层面的文化不可翻译性（参考维特根斯坦"不可说，则沉默"）

**冲突3：理想 vs 现实**
- 五层定义的是理想的原则，实际执行可能有偏差
- 权衡：定期审视和更新你的Skill，反映现实变化

## 04 VALIDATING — 验证标准

### 测试用例1：清晰性测试

**测试**：随机选择5个人，给他们DEFINING和INSTANTIATING，问"这个原则是什么？"

**通过标准**：
- ✅ 至少4个人（80%）给出相似的理解
- ✅ 理解中包含了核心价值观，不仅仅是规则

**失败示例**：
```
DEFINING: "AI应该有同情心"
随机5人的理解：
1. AI不应该拒绝用户 ❌ (太狭隘)
2. AI应该有感情 ❌ (理解错了)
3. AI应该理解用户的困难 ✓
4. AI不能冷漠 ❌ (反面表述)
5. AI应该考虑用户感受 ✓
通过率：40% → 需要重写DEFINING
```

### 测试用例2：具体性测试

**测试**：INSTANTIATING中的例子是否完美地体现了DEFINING？

**通过标准**：
- ✅ 每个好例子都清晰展现原则
- ✅ 每个坏例子都违反原则的同一个方面
- ✅ 没有歧义的例子

**失败示例**：
```
DEFINING: "设计应该尊重用户认知能力"
INSTANTIATING:
好例子："Apple的极简设计"  ✓ (清晰)
坏例子："过度装饰的网站"   ✓ (清晰)
模糊例子："Windows的设计"  ❌ (有些好，有些不好)
→ 需要用更明确的例子
```

### 测试用例3：FENCING清晰性测试

**测试**：给一个"灰色地带"的情况，问"这个Skill应不应该应用？"

**通过标准**：
- ✅ 能清晰解释为什么应用或不应用
- ✅ FENCING中提到的因素都用上了

**例子**：
```
SKILL: "代码应该为人类而写"
灰色地带：机器学习模型的代码，通常很复杂且难以理解

应该应用吗？
答案：应用，但要调整。因为：
- 团队需要理解模型的逻辑（FENCING的"团队项目"适用）
- 但可能不需要100%的可读性（因为数学复杂性是必需的）
→ 这说明FENCING定义清晰
```

### 测试用例4：跨文化理解测试

**测试**：请至少3种文化背景的人来理解CONTEXTUALIZING

**通过标准**：
- ✅ 不同文化的人都认为CONTEXTUALIZING尊重了他们的背景
- ✅ 核心原则在不同文化中仍然可识别
- ✅ 没有强行翻译，而是真正的转化

**失败示例**：
```
SKILL: "沉默的艺术"
CONTEXTUALIZING:
中文版：强调"陪伴" ✓
日文版：强调"間" ✓
英文版："just don't talk"  ❌ (太简化，丢失了核心)
→ 英文版需要重新写
```

### 测试用例5：可验证性测试

**测试**：能否设计明确的指标来验证这个Skill是否被遵循？

**通过标准**：
- ✅ 至少3个可测量的指标
- ✅ 指标不依赖主观判断
- ✅ 可以自动化或由第三方验证

**例子**：
```
SKILL: "代码应该为人类而写"
可验证的指标：
1. 函数长度不超过50行 (可自动化检查)
2. 变量名长度至少5个字符 (可自动化检查)
3. 注释行数 > 代码行数的10% (可自动化检查)
4. 新员工30分钟内的理解度 (需要用户研究)
✓ 全部可验证
```

## 05 CONTEXTUALIZING — 文化适配

### 中文语境版本

**原则名称**：将自然语言转化为可信的技能

**核心调整**：
- 中文强调"整体性"和"和谐"
- 五层架构可以被理解为"道"（DEFINING）→ "术"（后四层）的关系
- FENCING的"冲突"可以用"阴阳"的平衡来理解

**具体建议**：
```
【DEFINING】(道)
用价值观而不是规则表述，强调"无为而治"的境界

【INSTANTIATING】(象)
用具体的、日常的例子，而不是极端情况

【FENCING】(界)
强调边界的流动性，而不是绝对划分
中文表述："在什么样的春风下，这朵花会绽放？"

【VALIDATING】(验)
邀请共识而不是绝对验证
中文表述："多少同道者认可这个原则？"

【CONTEXTUALIZING】(化)
强调文化的代际传承
中文表述："这个原则如何在我们的文化里世代相传？"
```

### 英文语境版本

**原则名称**：Transform Natural Language into Trustworthy Skills

**核心调整**：
- 英文强调"个体性"和"清晰性"
- 用逻辑论证而不是感悟
- VALIDATING强调"证据"和"可重复性"

**具体建议**：
```
【DEFINING】(Thesis)
清晰的单句陈述，可以被反驳

【INSTANTIATING】(Evidence)
用科学案例和研究支持

【FENCING】(Boundaries)
用"if-then"的逻辑明确边界

【VALIDATING】(Verification)
强调"可重复"和"可审计"

【CONTEXTUALIZING】(Adaptation)
描述不同文化的"变体"，而不是"演化"
```

### 日文语境版本

**原則名称**：自然言語を信頼できるスキルに変換する

**核心调整**：
- 日文强调"美学"和"余白（間）"
- 重视"察する"（察觉/感受）
- 接受某些层面的"言葉にならない"（无法言说）

**具体建议**：
```
【DEFINING】(本質)
原則の本質を捉える、余白を持つ表現

【INSTANTIATING】(例)
日本文化の例、茶道や武道から借用

【FENCING】(場)
「場合」によって変わることを認める
西洋的な絶対的境界ではなく「空気を読む」

【VALIDATING】(確認)
數値的な検証より、「多くの人が感じる」ことを重視

【CONTEXTUALIZING】(深化)
文化の層を尊重し、翻訳不可能な部分を認める
（維特根斯坦の「言えぬことには沈黙しなければならぬ」）
```

## 📖 使用这个Skill的方式

### 对个人：
如果你有一个价值观想法，按照这个Skill的INSTANTIATING部分的三个案例来做：
1. 从DEFINING开始
2. 写出3-5个INSTANTIATING例子
3. 定义FENCING边界
4. 创建VALIDATING测试用例
5. 添加CONTEXTUALIZING（至少中英文）

### 对团队：
使用VALIDATING部分的测试用例来验证团队的价值观是否被清晰传达和理解。
